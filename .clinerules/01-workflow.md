# Cline 작업 규칙

## 1. 작업 시작
- 한 번에 하나의 기능 단위만 작업한다.
- 파일을 생성하거나 수정하기 전에 현재 branch를 확인한다.
- 먼저 작업 목표, 변경 대상 파일, 구현 계획, 테스트 방법을 설명한다.
- 사용자의 승인 전에는 파일을 생성하거나 수정하지 않는다.
- 요구사항이 모호하거나 허용 파일 밖 수정이 필요하면 임의로 진행하지 말고 이유를 설명한다.

## 2. 변경 범위 통제
- 프롬프트에서 허용한 파일만 생성하거나 수정한다.
- 허용 범위 밖의 파일 수정이 필요하면 먼저 이유와 대상 파일을 보고하고 추가 승인을 기다린다.
- 기존 기능과 데이터 구조를 유지하고, 요청하지 않은 기능을 미리 구현하지 않는다.
- 불필요한 리팩터링, 파일명 변경, 함수명 변경, 디렉터리 구조 변경을 하지 않는다.

## 3. Git 작업 제한
- 다음과 같은 상태 확인 목적의 읽기 전용 Git 명령만 실행하거나 제안할 수 있다.
  - `git status`
  - `git status --short`
  - `git diff`
  - `git diff --stat`
  - `git diff --staged`
  - `git branch --show-current`
  - `git log --oneline`
  - `git remote -v`
- 다음과 같이 Git 상태를 변경하는 명령은 사용자가 직접 수행한다.
  - `git add`
  - `git commit`
  - `git push`
  - `git pull`
  - `git merge`
  - `git reset`
  - `git restore`
  - `git clean`
  - branch 생성·삭제·전환
- Git 상태를 변경하는 명령은 사용자의 명시적 요청 없이는 실행하지 않는다.

## 4. 실행·테스트 명령
- `python app.py`, `python -m unittest discover -s tests -v`처럼 실행 결과를 확인하기 위한 명령은 사용자 승인 후 실행할 수 있다.
- 패키지 설치, 파일 삭제, 대량 파일 변경, Git 상태 변경과 관련된 터미널 명령은 사용자의 명시적 승인 없이는 실행하지 않는다.
- 테스트 실행 전에는 실행할 명령을 먼저 설명한다.

## 5. 작업 완료 보고
- 작업 후 실행 방법과 테스트 방법을 안내한다.
- 가능한 경우 unittest를 실행하고 결과를 보고한다.
- 마지막에는 `git status`와 `git diff` 기준으로 다음 내용을 정리한다.
  1. 생성하거나 수정한 파일
  2. 핵심 변경 내용
  3. 실행한 테스트와 결과
  4. 남아 있는 위험 요소 또는 확인이 필요한 사항
